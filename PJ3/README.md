# PJ3 Blender 3D 建模系统

PJ3 是一个**面向中式传统家具的多智能体 3D 建模系统**。它把一句自然语言描述（中/英文）变成可在 Blender 中打开的三维模型。PJ3 内部由一组分工协作的 LLM 智能体完成 **规划 → 写 Blender 代码 → 执行渲染 → 批判 → 修复** 的闭环迭代，并支持人在环中随时介入掌舵。擅长官帽椅、案几、八仙桌、屏风、圈椅等明式家具。

---

## 目录结构

```
PJ3/
├── main.py              # 流水线编排 + 交互/自动循环
├── agents/              # planner / api_rag / coder / fixer / geom_verifier / vlm_critic / ui
├── blender/             # runner（无头执行）+ render（4 视角渲染）
├── prompts/             # 各 agent 的 system/user 提示词
├── knowledge/           # bpy_api.jsonl（内省本机 Blender 得到的 API 文档库）
├── build_bpy_kb.py      # 构建知识库
└── outputs/             # 每次运行产物
```

## 安装与使用

1. 装依赖 + Blender
```bash
pip install -r requirements.txt
# 安装 Blender 4.x/5.x，让 `blender` 在 PATH 中，或：
export BLENDER_PATH=/path/to/blender
```

2. 配密钥（项目根目录 `.env`）
```
KIMI_API_KEY=...        # Planner/Coder/Fixer/VLM 主模型
GLM_API_KEY=...         # VLM 回退（可选）
OUTPUT_ROOT=outputs     # 可选，默认 outputs/
```

3. 构建 bpy API 知识库（首次 / 换 Blender 版本时跑一次）
```bash
blender --background --python build_bpy_kb.py   # 生成 knowledge/bpy_api.jsonl
```

4. 运行
```bash
# 交互式（默认）：每轮暂停复审，启用几何验证 + VLM
python main.py "做一张明式官帽椅" --vlm --geom

# 自动/批处理：最多迭代 N 轮
python main.py "长凳" --auto --iterations 2 --vlm --geom

# 其它：--no-planner 跳过规划  ·  --verbose 打印 agent 完整对话
```

每轮交互菜单：`[a]` 接受结束 · `[f]` 修复检测到的问题 · `[c]` 输入自己的意见（自动归类路由）· `[q]` 退出。

## 核心思想

把"3D 建模"当成**代码生成 + 程序化校验的闭环**，而非端到端生成网格：

- **代码即表示**：Blender Python（`bpy`）可执行、可读、可增量修改，天然适合 LLM 生成与定向修复。
- **分而治之**：先规划出结构化零件图，再逐件写码，比"一句话直接出码"更可控、不易漏件。
- **双重校验**：渲染图给 VLM 看（外观/比例），同时在 Blender 内跑确定性几何校验（VLM 看不见的装配 bug：空场景、尺度爆炸、连接断开、严重穿模）。
- **结构化修复**：每个问题带 `kind`（结构性 / 参数性），分别路由到"改计划+重构"或"只调数值"，互不越界。

## 系统组件

- **Planner**：将自然语言转换为结构化 JSON 计划；并负责在批判之后**重计划**。

- **API-RAG**：检索本地 Blender 真实 API 文档。写码前按构造方式取，崩溃时按 traceback 取精确签名/socket。

- **Coder**：计划 + API 文档 → 可执行 `bpy` 代码，并拦截已知崩溃模式。

- **Runner / Render**：把代码 + 几何校验 + 渲染片段拼成脚本， 无头执行；渲染 4 视角并自动取景。

- **几何验证器**：作为 `bpy` 片段注入到 Blender 进程内，输出 JSON 问题，只报零误报硬错误。

- **VLM Critic**：把 4 张渲染图喂给视觉模型返回外观问题。

- **Fixer**：`fix_parameters`（只调数值）与 `fix_structure`（修改结构和组件）。

- **命令行交互**：串起整条流水线 + 交互/自动两种循环。

模型：Planner/Coder/Fixer/VLM 默认 Kimi k2.5（Moonshot，OpenAI 兼容接口）；检索用 `bge-small-en-v1.5` 句向量（离线、磁盘缓存）。

## 构造方式

每个零件不从封闭形状枚举里选，而描述它是怎么造出来的（4 种 + 自然语言轮廓 `profile_desc`）：

- `primitive` 规则实体/纯矩形板（直腿、座面、圆盘）
- `extrude` 平的非矩形轮廓（S 形靠背板、壸门牙板、券口）
- `revolve` 旋转对称体（花瓶、鼓腿、灯罩）
- `sweep` 沿路径扫掠的弯杆（搭脑、扶手、圈椅圈）

> 枚举是打地鼠——每出现一个表达不了的零件就崩。4 种构造操作几乎覆盖所有家具零件。

## 与 LL3M 的对比与创新

本项目参考了 [LL3M 项目](https://threedle.github.io/ll3m/)。LL3M 的迭代回路本质是 **"写代码，看渲染图，改代码"**。本项目沿用了它"代码即建模 + API 文档检索（写码前/报错时两阶段）"的主线，主要在**校验**和**修复**两端做了扩展。

1. **计划驱动的连接校验（把"不许悬空"变成可机检约束）**

LL3M **没有几何检查器**。本项目在 Blender 进程内注入一段 `bpy` 校验代码（与渲染同进程，能读到求值后的真实网格），专查四类**几何硬错**：空场景/空网格、零件尺度退化或整体爆炸、**声明的连接断开**（间距 > 1cm）、**非预期穿模**（未声明相连的两件真实网格 BVH 相交且重叠 ≥ 较小件体积的 30%）。关键一步是把计划的 `connections` **确定性地注入**为模块级 `CONNECTIONS` 常量（而非指望 Coder 自己写对——它经常漏写，会让连接检查静默失效）。

在缺少几何检查器的情况下，建模出来的家具模型往往时零散不相连的（零件悬空、整体尺度离谱、部件互相穿插），需要多次迭代才能完善。

2. **几何验证与 VLM 双轨校验，各管一摊**

LL3M 的校验只有"看图"这一种途径，但看图时大模型对几何会**幻觉**。例如：某次台灯生成里 VLM 报"灯杆分段、段间有可见缝隙"，可代码里灯杆是**一根连续圆柱**、外面只套了 3 个装饰环，同一轮几何验证报 0 问题——纯属看图幻觉。所以连接、缝隙、穿模、尺度这类**有客观真值**的判断，交给看图模型并不可靠。本项目据此把校验拆成正交的两轨：

- **几何验证（读真实坐标，零误报）** 负责一切几何事实：空场景、尺度退化/爆炸、连接是否真的接上、是否穿模。它有 ground truth，但说不出"好不好看"。
- **VLM（看图）** 只负责几何验证看不到的：整体比例与视觉平衡、风格是否到位、设计是否单调、组件缺失/冗余、对称性、材质配色，以及一眼可见的**宏观摆放错误**（如灯泡露在灯罩外）。它被**明确禁止**报缝隙/接触/穿模/绝对尺度。

两者职责互斥、能力互补：VLM 管"长得像不像、好不好看"，几何验证管"装没装对"。

3. **结构化 Issue 路由 + 双职责修复（结构问题必须先改计划）**

LL3M 是单一精修回路、只改代码。本项目给每个问题打上 `kind`（结构性 / 参数性），结构性走 `replan + fix_structure`（先修订计划，再按 diff 增删/重塑零件），参数性走 `fix_parameters`（只调数值、禁止增删件）。

为什么结构问题**一定要先改计划、而不能只改代码**：缺件或结构错时若只在代码上打补丁、不更新 plan，迭代几轮后 Fixer 失去了结构蓝图，对着一份和代码已经对不上的旧计划越改越离谱；更糟的是注入给几何验证的 `CONNECTIONS` 来自 plan，plan 不更新，新增/改动的连接关系几何验证**根本检查不出来**——校验跟着一起失明。所以结构轮先 `replan` 刷新蓝图，再让 Fixer 在新蓝图（`plan_spec`）下重构，几何验证的连接表也随之同步。两条路都携带本轮全部问题、绝不丢参数问题（曾因结构轮丢掉参数问题，导致几何断连永远修不上、迭代不收敛）。

4. **人在环中为默认**

LL3M 把用户引导精修作为最后一个阶段。本项目把交互式设为默认：每轮渲染+批判后暂停，展示结果，让用户选择 `[a]` 接受 / `[f]` 修检测到的问题 / `[c]` 提自己的意见 / `[q]` 退出；自由意见会被自动归类为结构/参数并路由到对应修复路径。`--auto` 才是非交互的批处理/评测模式。让用户随时掌舵，而不是看完一长串自动迭代再说。

5. **中式家具领域专精**

Planner 的 system prompt 内置了中式家具的参考尺寸（案几/八仙桌/官帽椅/圈椅……）、榫卯/牙板/搭脑/券口等术语、以及传统比例规则（如腿高/总高 ≈ 0.55–0.65），并要求按"工匠会分开做的结构子件"来分解，避免把一件家具糊成三个方块。

6. **本地 API 文档库 + 双阶段检索（写码前按构造方式取、崩溃时按 traceback 取精确签名/socket）**

`bpy` 的 API 跨版本变动很大（最典型的是 4.x→5.x 的 Principled BSDF socket 改名），大模型记忆里的签名往往是旧的或干脆是幻觉，是头号崩溃来源。本项目不单纯依赖大模型的内存，而是用 `build_bpy_kb.py` **内省本机安装的 Blender**（RNA introspection）离线生成一份与本地版本完全对齐的 API 文档库（`knowledge/bpy_api.jsonl`），用 `bge-small-en-v1.5` 句向量检索（离线、磁盘缓存，无句向量时退化为关键词重叠）。检索在两个阶段介入：

- **写码前（按构造方式取）**：`retrieve_for_plan` 按计划里每个零件声明的**构造算子**（extrude / revolve / sweep / primitive）精确取对应的数据 API 文档面，只为计划真正用到的高级构造拉文档（trivial 的 cube/cylinder 不拉、避免噪声），并**始终注入版本正确的 Principled BSDF socket 清单**。
- **崩溃时（按 traceback 取）**：`retrieve_for_error` 把噪声 traceback 蒸馏成定位失败 API 的信号——放大引号里的 socket 名（`"Specular"`）、`bpy.*` 点路径、关键字参数（`radius1=`），再用**关键词 + 语义混合排序**把精确签名/socket 喂回 Fixer。

这样代码生成与崩溃修复都**锚定在本机真实文档**上，而不是赌大模型记得对，显著减少了幻觉式 API 调用与反复试错。

## 已知局限

- **几何验证只报零误报硬错误**（空场景 / 尺度退化爆炸 / 声明连接 >1cm 断开 / 未声明却 ≥30% 体积嵌入）；关节处的**小穿模几何不可证**。
- **后期无效轮次**：循环缺"已收敛即停"判据，迭代后期可能产出肉眼看不出的厘米级微调。
- **VLM 分辨力**：纯灰模型、固定视角下难分辨厘米级差异。
- **依赖外部 API**：四个 agent 都走云端模型，需密钥与网络。
